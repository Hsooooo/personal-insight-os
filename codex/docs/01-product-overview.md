# 제품 개요

## 문서 목적

Personal Insight OS의 제품 정의, 핵심 가치, MVP 판단 기준을 정리합니다. 이 문서는 기능 구현보다 먼저 제품이 무엇을 해결해야 하는지 합의하기 위한 기준 문서입니다.

## 제품 한 줄 정의

나의 건강, 운동, 기록, 생각, 행동 데이터를 수집하고, 이를 도메인별 노드와 관계 그래프로 구조화한 뒤, LLM을 통해 근거 기반 개인화 인사이트를 제공하는 웹서비스입니다.

## 제품형 설명

흩어진 개인 데이터를 하나의 지식 그래프로 연결해 "내가 왜 이런 상태인지"를 설명해주는 Personal Insight OS입니다.

이 서비스는 단순한 Garmin 대시보드가 아닙니다. Garmin은 첫 번째 데이터 소스일 뿐이며, 핵심은 개인 데이터를 도메인화하고 관계로 연결해 해석 가능한 자기이해 시스템을 만드는 것입니다.

## 핵심 가치

기존 건강/운동 앱은 데이터를 보여주는 데 집중합니다.

Personal Insight OS는 데이터를 연결하고 해석하는 데 집중합니다.

단순 표시 예시는 다음과 같습니다.

```text
오늘 수면 6시간 10분
러닝 8.2km
평균 심박 152
스트레스 높음
```

이 제품이 답해야 하는 질문은 다음에 가깝습니다.

```text
최근 컨디션이 떨어진 이유는 무엇인가?
러닝 기록이 좋았던 날들의 공통점은 무엇인가?
수면 부족이 내 운동 성과에 어떤 영향을 주는가?
최근 내 생활 패턴에서 위험 신호가 보이는 부분은 무엇인가?
내 목표와 실제 행동은 얼마나 일치하고 있는가?
```

## 제품이 지향하는 경험

사용자는 여러 화면에서 숫자를 해석하기 위해 직접 맥락을 조합하지 않아도 됩니다. 시스템은 수면, 운동, 건강 지표, 목표, 인사이트를 하나의 개인 그래프에 연결하고, 질문이 들어오면 실제 데이터 근거를 우선 수집한 뒤 답변합니다.

좋은 답변은 다음 조건을 만족해야 합니다.

```text
1. 결론이 명확하다.
2. 실제 조회된 근거가 함께 표시된다.
3. 인과관계를 단정하지 않는다.
4. 사용자의 피드백을 통해 이후 품질이 개선된다.
5. 의료 진단처럼 보이는 표현을 피한다.
```

## MVP 성공 기준

MVP는 다음 질문에 답할 수 있으면 성공입니다.

```text
Garmin 데이터를 수집해서 내 운동/수면/건강 지표를 조회할 수 있는가?
수집된 데이터를 Activity, Sleep, HealthMetric, Date, Insight 같은 도메인으로 변환할 수 있는가?
PostgreSQL에는 원천/정형 데이터를 저장하고, Neo4j에는 의미 있는 관계를 저장할 수 있는가?
사용자가 자연어로 질문했을 때, 실제 데이터 근거를 기반으로 답변할 수 있는가?
생성된 인사이트를 저장하고, 사용자가 맞음/틀림/애매함으로 피드백할 수 있는가?
```

## 핵심 설계 원칙

```text
Raw 데이터는 버리지 않는다.
정형 데이터는 PostgreSQL에 저장한다.
의미 관계는 Neo4j에 저장한다.
LLM 답변은 evidence-first 방식으로 생성한다.
인사이트는 일회성 응답이 아니라 누적되는 개인 지식으로 취급한다.
```

## 현재 단계의 결론

이 프로젝트의 핵심은 Garmin 연동 자체가 아닙니다.

Garmin은 첫 번째 개인 데이터 소스입니다. 진짜 핵심은 다음입니다.

```text
개인의 데이터를 도메인화한다.
도메인화된 정보를 노드로 만든다.
노드 간 관계를 구성한다.
PostgreSQL에는 원천과 정형 데이터를 저장한다.
Neo4j에는 의미 관계를 저장한다.
LLM은 RAG를 통해 근거 기반으로 개인화 인사이트를 생성한다.
사용자는 인사이트를 저장하고 교정한다.
```

따라서 이 서비스는 단순 건강 대시보드가 아니라, 개인의 몸과 행동을 이해하기 위한 그래프 기반 자기이해 시스템입니다.
